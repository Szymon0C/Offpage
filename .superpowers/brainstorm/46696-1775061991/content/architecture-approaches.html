Architektura Offpage — 3 podejścia

Jak zorganizować frontend Tauri + edytor + AI inference

A

Monolityczny SPA — Rekomendowane

Jeden frontend app (React/Solid) z webview jako edytorem. Chat, WYSIWYG, podgląd i inline prompt — wszystko w jednym oknie z panelami.

Pros

  • Najprostszy do zbudowania i utrzymania
  • Łatwy state management — jeden kontekst
  • Szybka komunikacja między panelami
  • Jeden build pipeline

Cons

  • Edytor WYSIWYG w tym samym webview co app — potencjalne konflikty CSS
  • Trudniejsza izolacja podglądu strony od UI aplikacji
B

Dual Webview — App + iframe Preview

Główny webview to UI aplikacji (chat, panel szablonów, ustawienia). Podgląd strony w izolowanym iframe. WYSIWYG i inline prompt operują na iframe.

Pros

  • Pełna izolacja CSS/JS strony od UI apki
  • Podgląd strony wygląda dokładnie jak na produkcji
  • WYSIWYG naturalnie operuje na rzeczywistym DOM strony

Cons

  • Komunikacja app↔iframe wymaga postMessage API
  • Nieco więcej złożoności architektonicznej
C

Multi-window — Osobne okna Tauri

Tauri pozwala na wiele okien. Jedno okno = edytor/chat, drugie okno = live preview strony. Komunikacja przez Tauri IPC.

Pros

  • Maksymalna izolacja
  • Użytkownik może ustawić preview na drugim monitorze
  • Każde okno ma pełny viewport

Cons

  • Złożone zarządzanie stanem między oknami
  • UX mniej intuicyjny dla non-technical users
  • Więcej kodu do utrzymania
