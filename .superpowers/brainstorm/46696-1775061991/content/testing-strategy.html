Sekcja 8: Strategia testowania

Co testujemy, jak, i na jakim poziomie

Piramida testów
E2E
Playwright · najmniej
Integracyjne
Vitest + Tauri test utils · średnio
Unit
Vitest (frontend) + cargo test (Rust) · najwięcej
Unit testy
Frontend (Vitest):
• Zustand stores — state transitions
• Prompt builder — construction logiki promptów
• HTML validator — detekcja broken HTML
• postMessage serialization
Rust (cargo test):
• File manager — read/write/export
• Sidecar manager — config, health check
• Deploy service — request building
• SQLite queries — CRUD projekty
Integracyjne
Frontend ↔ Rust IPC:
• Invoke commands i poprawne response
• Event streaming (AI tokens)
App ↔ iframe:
• postMessage protokół
• Section detection + selection
• WYSIWYG edits propagacja
AI pipeline:
• Prompt → llama-server → valid HTML
• Snapshot creation po edycji
E2E
Playwright + Tauri driver:
• Pełny flow: prompt → generate → preview
• Template select → customize → preview
• Chat edit → strona się zmienia
• Inline edit → sekcja się zmienia
• WYSIWYG edit tekstu
• Undo/redo historia
• Export plik HTML
Uwaga: E2E z AI są wolne i niedeterministyczne. Mocki llama-server dla CI, prawdziwy model dla smoke testów.

Narzędzia i CI

  • Vitest — unit + integracyjne frontend, szybki, natywny ESM, kompatybilny z Vite
  • cargo test — unit Rust backend
  • Playwright — E2E z Tauri WebDriver
  • GitHub Actions CI — unit + integracyjne na każdy PR, E2E nightly
  • Mock llama-server — lekki HTTP server zwracający predefiniowany HTML dla testów CI (bez GPU)
