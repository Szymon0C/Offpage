Sekcja 4: Model danych i persystencja

SQLite — co przechowujemy i jak

Schemat bazy danych (SQLite)
📁 projects
id TEXT PRIMARY KEY
name TEXT — "Kawiarnia Landing Page"
html TEXT — aktualny HTML strony
template_id TEXT? — jeśli z szablonu
site_type TEXT — landing/portfolio/blog/ecommerce
deploy_config JSON? — {provider, site_id, url}
created_at TIMESTAMP
updated_at TIMESTAMP
📸 snapshots (undo/redo)
id TEXT PK
project_id TEXT FK → projects
html TEXT — snapshot strony
description TEXT — "Zmieniono nagłówek"
version INTEGER — kolejność
created_at TIMESTAMP
💬 chat_messages
id TEXT PK
project_id TEXT FK → projects
role TEXT — user/assistant
content TEXT — treść wiadomości
edit_type TEXT? — chat/inline/wysiwyg
created_at TIMESTAMP
🎨 templates
id TEXT PK
name TEXT — "Minimal Portfolio"
category TEXT — landing/portfolio/...
html TEXT — HTML szablonu
thumbnail BLOB — podgląd
version TEXT — "1.0.0"
⚙️ settings
key TEXT PK
value JSON
— hardware_tier: "recommended"
— quantization: "Q4_K_M"
— deploy_tokens: {netlify: "...", ...}
— theme: "dark"

Dlaczego SQLite

  • Lokalna app = lokalna baza — zero serwera, zero konfiguracji
  • Jeden plik — łatwy backup, przenoszenie między maszynami
  • Tauri ma natywne wsparcie — plugin tauri-plugin-sql
  • Snapshots jako undo/redo — każda zmiana AI = nowy snapshot, użytkownik nawiguje historią wersji
