Sekcja 7: Error handling i edge cases

Co może pójść nie tak i jak to obsługujemy

Scenariusze błędów
🤖 AI / llama-server
Sidecar nie startuje
→ Sprawdź wymagania sprzętowe, pokaż user-friendly komunikat z linkiem do FAQ. Zaproponuj mniejszy model (3B) jeśli za mało RAM.
Generowanie się zawiesza
→ Timeout po 120s. Pokaż "Generowanie trwa zbyt długo" + przycisk "Spróbuj ponownie". Zachowaj ostatni dobry stan.
Zepsuty HTML output
→ Walidacja HTML przed wstrzyknięciem do iframe. Jeśli broken — rollback do ostatniego snapshota, info w chacie "Coś poszło nie tak, wracam do poprzedniej wersji".
Context window exceeded
→ Automatyczne przycinanie historii chatu. Dla bardzo dużych stron — ostrzeżenie i sugestia użycia inline edit zamiast pełnego chat edit.
🚀 Deploy
Brak internetu
→ Wykryj offline przed deploy. Komunikat "Brak połączenia — eksportuj plik lokalnie i opublikuj później".
Token wygasł
→ Automatyczny re-auth flow. Jeśli się nie uda — "Zaloguj się ponownie do [provider]".
API limit / błąd
→ Retry z exponential backoff (max 3x). Jeśli nadal fail — pokaż błąd providera + opcja eksportu lokalnego.
💾 Aplikacja
Crash / kill procesu
→ Auto-save co 30s do SQLite. Przy starcie — wykryj niezapisane zmiany, zaproponuj recovery.
Brak miejsca na dysku
→ Sprawdź przed download modelu. Komunikat z wymaganą ilością miejsca.
Pierwsza instalacja
→ Onboarding wizard: pobierz model (~4GB), wykryj sprzęt, ustaw optymalną konfigurację. Progress bar z ETA.

Zasady ogólne

  • Nigdy nie trać pracy użytkownika — auto-save + snapshots = zawsze można wrócić
  • User-friendly komunikaty — zero stack trace'ów, prosty język, jasna akcja do podjęcia
  • Graceful degradation — jeśli AI zawiedzie, daj opcję manualną (eksport, edycja, retry)
  • Onboarding — pierwszy launch = wizard który prowadzi przez setup (model download, hardware detection)
