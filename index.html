Setup od zera

Kompletna instrukcja — od instalacji Claude Code do pełnego setupu z pluginami, MCP serverami i konfiguracją projektu. Zajmuje ~10 minut.

1. Instalacja Claude Code

Terminal
# Instalacja przez npm (wymaga Node.js 18+)
$ npm install -g @anthropic-ai/claude-code
 
# Pierwsze uruchomienie — logowanie do Anthropic
$ claude
Welcome to Claude Code! Please log in...
 
# Lub uruchomienie w konkretnym projekcie
$ cd ~/projects/my-app && claude
Dostępne platformy: CLI w terminalu, desktop app (Mac/Windows), web app (claude.ai/code), rozszerzenia IDE (VS Code, JetBrains). Wszystkie współdzielą ten sam silnik.

2. Wybór modelu

W ~/.claude/settings.json ustawiamy domyślny model:

settings.json { "model": "opus" } // Dostępne modele: // "opus" — Claude Opus 4.6 (najpotężniejszy, nasz default) // "sonnet" — Claude Sonnet 4.6 (szybszy, tańszy) // "haiku" — Claude Haiku 4.5 (najszybszy, najtańszy) // W sesji: /fast toggle szybkiego output (ten sam model)

4. Superpowers — 14 skills w szczegółach

Plugin superpowers to serce naszego setupu. Dodaje 14 wyspecjalizowanych skills, które Claude automatycznie wywołuje gdy pasują do zadania:

B

brainstorming

Wywoływany przed każdą implementacją. Prowadzi dialog pytanie-odpowiedź, buduje design, czeka na Twoje zatwierdzenie. Hard gate: nie napisze ani linii kodu bez Twojego OK.

Automatyczny
T

test-driven-development

Napisz test → popatrz jak failuje → napisz minimalny kod. Claude nie pisze implementacji przed testem. Rigid — nie daje się "obejść".

Rigid
D

systematic-debugging

Przed losowym fixem — znajdź root cause. Symptom fix = failure. Claude buduje hipotezy, testuje je, dopiero potem naprawia. Używany przy każdym bugu.

Rigid
P

writing-plans

Tworzy plany implementacji w Plan Mode. Identyfikuje krytyczne pliki, zależności, ryzyka. Plan jest widoczny i edytowalny.

Plan Mode
E

executing-plans

Realizuje zatwierdzony plan krok po kroku. Śledzi postęp w TodoWrite, oznacza ukończone zadania.

Po planowaniu
CR

requesting-code-review

Wysyła kod do review po ukończeniu kroku. Uruchamia agenta code-reviewer w tle.

Po implementacji
RR

receiving-code-review

Przetwarza feedback z code review — naprawia znalezione problemy, iteruje aż review przejdzie.

Po review
V

verification-before-completion

Przed deklaracją "done" — Claude sprawdza czy build przechodzi, testy pass, typy OK. Nie deklaruje sukcesu bez weryfikacji.

Automatyczny
PA

dispatching-parallel-agents

Wysyła wiele agentów równolegle — np. review + testy + explore jednocześnie.

Optymalizacja
SD

subagent-driven-development

Deleguje zadania do wyspecjalizowanych subagentów zamiast robić wszystko w głównym kontekście.

Architektura
GW

using-git-worktrees

Izolowane kopie repo przez git worktree. Agent pracuje w worktree, nie wpływa na główny branch.

Git
FB

finishing-a-development-branch

Zamykanie brancha — commit, push, PR, merge. Cały flow w jednym kroku.

Git
WS

writing-skills

Meta-skill — pomaga tworzyć nowe custom skills dla Twojego zespołu/projektu.

Meta
US

using-superpowers

Skill startowy — uczy Claude jak znajdować i używać inne skills. Ładowany na początku każdej sesji.

Bootstrap
Kluczowe: Superpowers działa automatycznie — nie musisz wywoływać skills ręcznie. Claude sam rozpoznaje kiedy brainstorming, debugging czy TDD jest potrzebny i aktywuje odpowiedni skill. Jeśli jest choćby 1% szans że skill pasuje, zostanie użyty.

Fundamenty

Narzędzia wbudowane

Claude Code ma dedykowane narzędzia do każdego typu operacji. Kluczowa zasada: nigdy nie używaj Bash do rzeczy, które mają dedykowane narzędzie.

R

Read

Czytanie plików. Zawsze czytaj plik przed edycją. Obsługuje obrazy, PDF-y, Jupyter notebooks. Można czytać zakres linii (offset + limit) dla dużych plików.

Zamiast: cat, head, tail
E

Edit

Precyzyjne edycje — podajesz old_string i new_string. Wysyła tylko diff, nie cały plik. Parametr replace_all do masowej zamiany (np. rename zmiennej).

Zamiast: sed, awk
W

Write

Tworzenie nowych plików lub pełne przepisanie istniejących. Wymaga wcześniejszego Read dla istniejących plików. Preferuj Edit gdy zmieniasz fragment.

Zamiast: echo >, heredoc
G

Grep

Wyszukiwanie w treści plików — regex, filtrowanie po typie pliku, tryby: content, files_with_matches, count. Obsługuje multiline matching.

Zamiast: grep, rg
*

Glob

Wyszukiwanie plików po wzorcu nazwy: **/*.tsx, src/stores/*.ts. Sortuje po dacie modyfikacji. Idealne do szukania konkretnych plików.

Zamiast: find, ls
$

Bash

Komendy systemowe: git, npm, cargo build, testy. Użwaj tylko gdy brak dedykowanego narzędzia. Obsługuje timeout i run_in_background.

Ostateczność
T

TodoWrite

Lista zadań w sesji. Statusy: pending → in_progress → completed. Pomaga śledzić postęp przy złożonych zadaniach. Max 1 task in_progress.

Organizacja pracy
Realne użycie z projektu: W Offpage każda sesja zaczynała się od Read kluczowych plików (store'y, komponenty), Grep do znalezienia zależności, a Edit do chirurgicznych zmian. Write był używany tylko przy tworzeniu nowych komponentów lub pełnym przepisaniu promptów AI.
Równoległość

System agentów (subagents)

Agenci to niezależne procesy, które mogą pracować równolegle. Każdy ma swój kontekst i dostęp do narzędzi. Kluczowe: uruchamiaj niezależnych agentów równolegle.

E

Explore

Szybkie eksplorowanie codebase. Trzy tryby: quick, medium, very thorough. Używaj do szerokiego rozpoznania — np. "jak działają deploy providery?"

P

Plan

Architekt software'u — projektuje plany implementacji. Zwraca kroki, identyfikuje krytyczne pliki, rozważa trade-offy. Nie pisze kodu.

CR

code-reviewer

Review po ukończeniu większego kroku. Sprawdza zgodność z planem, standardy kodowania, potencjalne bugi. Używany po zakończeniu fazy.

GP

general-purpose

Uniwersalny agent do złożonych, wielokrokowych zadań. Szukanie kodu, odpowiadanie na pytania, badanie problemów. Ma dostęp do wszystkich narzędzi.

CS

code-simplifier

Upraszcza i refaktoryzuje kod. Skupia się na niedawno zmodyfikowanych plikach. Zachowuje funkcjonalność, poprawia czytelność.

Równoległe uruchomienie agentów

W Offpage używaliśmy agentów równolegle do review i testów:

Claude Code — parallel agents
// Claude wysyła JEDEN message z wieloma Agent tool calls:
 
Agent 1 → subagent_type: "code-reviewer"
"Review the Phase 5 deploy implementation against the plan"
 
Agent 2 → subagent_type: "Explore"
"Find all places where deploy_config is used"
 
Agent 3 → subagent_type: "general-purpose"
"Run TypeScript type-check and report errors"
 
// Wszystkie 3 pracują równolegle, wyniki wracają niezależnie
Worktree isolation: Agenty mogą pracować w izolowanym git worktree (isolation: "worktree"). Każdy dostaje kopię repo i nie wpływa na główny branch. Przydatne przy ryzykownych eksperymentach.
Proces

Workflow budowania Offpage

Tak wyglądał nasz cykl pracy. Każda faza miała plan, implementację, review i fixy.

1

Planowanie (EnterPlanMode)

Przed każdą fazą wchodziliśmy w Plan Mode — Claude analizuje codebase, proponuje architekturę, identyfikuje zależności. Plan jest widoczny i edytowalny. User akceptuje lub modyfikuje podejście.

2

Implementacja z TodoWrite

Plan rozbijany na zadania w TodoWrite. Claude pracuje sekwencyjnie — jedno zadanie na raz, mark as completed po zakończeniu. User widzi postęp w real-time.

3

Type-check & Build

Po implementacji: npx tsc --noEmit (TypeScript), cargo build (Rust). Fixy iteracyjne — Claude czyta błąd, naprawia, sprawdza ponownie.

4

Code Review (Agent)

Po ukończeniu fazy — code-reviewer agent sprawdza zgodność z planem. W przypadku Offpage dodatkowo CodeRabbit (GitHub bot) robił automated review na PR.

5

Git & PR

Commit z opisem zmian, push, PR do main. Claude generuje tytuł i body PR automatycznie (summary + test plan). Squash merge do main.

6

Manual testing & Fixy

User testuje ręcznie, raportuje problemy w naturalnym języku (nawet po polsku). Claude czyta opis, analizuje logi, identyfikuje root cause, naprawia.

Lekcja z projektu: Nie pomijaj manual testing. W Offpage automated review znalazł 14 bugów, ale prawdziwy blocker (brak dylibs dla llama-server) wyszedł dopiero przy ręcznym uruchomieniu. Claude potrafi czytać console logi wklejone do chatu i diagnozować root cause.

Rozszerzenia

Skills & Superpowers

Skills to wyspecjalizowane rozszerzenia Claude Code. Są wywoływane automatycznie lub ręcznie przez slash commands.

Claude analizuje staged changes, drafuje commit message ("why" nie "what"), dodaje Co-Authored-By. Weryfikuje czy nie commitujesz .env czy credentials. Respektuje pre-commit hooks.

# Wystarczy napisać: /commit # Claude automatycznie: 1. git status (sprawdza co się zmieniło) 2. git diff (analizuje zmiany) 3. git log (czyta styl commitów w repo) 4. Drafuje message + staging + commit

Wywoływany po ukończeniu większego kroku. Porównuje implementację z planem, szuka bugów, sprawdza standardy. W Offpage znalazł 14 bugów w jednym przebiegu.

Znalezione bugi w Offpage: Tauri invoke params (camelCase vs snake_case), stale closures w React useCallback, brakujące error handling, typ mismatch (string vs object) w SQLite, event listener leaks.

Skill automatycznie wywoływany przed EnterPlanMode. Pomaga przemyśleć podejście zanim zacznie się kodować. Rozważa alternatywy, identyfikuje ryzyka.

Wersjonowanie

Git workflow

Claude Code jest świadomy gita i potrafi zarządzać branchami, commitami i PR-ami. Ale domyślnie jest ostrożny — pyta przed destrukcyjnymi operacjami.

ZasadaDlaczego
Nigdy --force push bez potwierdzenia Może nadpisać cudze zmiany
Nigdy --no-verify Pomija hooks — diagnozuj przyczynę
Nowe commity, nie amend Amend po failed hook nadpisuje poprzedni commit
Nie commituje .env, credentials Bezpieczeństwo — ostrzega gdy próbujesz
Specific files, nie git add -A Zapobiega przypadkowemu dodaniu śmieci
Commituje TYLKO gdy poprosisz Nie jest nadmiernie proaktywny
PR creation flow # Claude automatycznie robi te kroki: 1. git status + git diff # co się zmieniło 2. git log + git diff main...HEAD # pełna historia brancha 3. Analiza WSZYSTKICH commitów # nie tylko ostatni! 4. Drafowanie title + body 5. gh pr create --title "..." --body "..." # via GitHub CLI # Format body: ## Summary - Bullet points zmian ## Test plan - [ ] Checklist do testowania

Sytuacje z Offpage

Rebase conflict po squash merge: Po squash merge PR do main, lokalny branch miał konflikty z rebase. Rozwiązanie: git reset --hard origin/main (z potwierdzeniem usera) i kontynuacja pracy na czystej bazie.
Worktree do izolacji: Claude Code pracował w git worktree (.claude/worktrees/gifted-satoshi) — izolowana kopia repo, nie wpływała na główny katalog. Worktree jest tworzony automatycznie.

Debugging session

i po chwili wraca do wyboru modelu — nic się nie dzieje"
 
Claude: // Dodaje debug logi z prefixami [AI], [ChatPanel]

1

Dodaj logi diagnostyczne

Dodaj console.log z prefiksami w kluczowych punktach flow. Np. [AI] Starting sidecar..., [ChatPanel] initAI...

2

User testuje i wkleja logi

Użytkownik uruchamia apkę, kopiuje output z console/devtools, wkleja do Claude Code.

3

Claude analizuje i naprawia

Claude czyta logi, identyfikuje root cause, naprawia problem. Weryfikuje fix przez type-check lub build.

Najlepsze praktyki

Tips & Tricks

Wnioski z ponad 6 faz rozwoju Offpage — co działa, czego unikać.

✓

Mów co chcesz, nie jak

"Napraw stylowanie sidebara — nie mieści się w 300px" jest lepsze niż "zmień padding z p-8 na p-3". Claude sam zbada i zaproponuje najlepszy fix.

✓

Pisz w swoim języku

Claude rozumie polski, angielski i inne. W Offpage komunikacja była po polsku, a kod i commity po angielsku — zero problemów.

✓

Wklejaj logi i screenshoty

Claude czyta pliki graficzne (Read tool). Console output, stack trace, screenshot błędu — im więcej kontekstu, tym szybsza diagnoza.

✓

Pozwól na autonomię

"Działaj jak uważasz" — Claude sam zdecyduje o kolejności, mergowaniu PR, strukturze. Możesz nadzorować lub dać pełną wolność.

✗

Nie powtarzaj instrukcji

Zapisz preferencje w CLAUDE.md lub pamięci. Nie powtarzaj "używaj TypeScript" w każdej sesji — raz wystarczy.

✗

Nie ignoruj review

Code-reviewer agent znalazł 14 bugów w Offpage naraz. Pomiń go a bugi trafią do produkcji.

✗

Nie batchuj na koniec

Commituj i pushuj regularnie. Jeden wielki commit na koniec = trudny review, konflikty, stracona praca przy crashu.

!

Context window się kończy

Przy dużych sesji Claude automatycznie kompresuje wcześniejsze wiadomości. Ważne informacje zapisuj do pamięci lub plików — nie polegaj na "pamiętaniu".
